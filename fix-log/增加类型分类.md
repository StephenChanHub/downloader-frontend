# 问：

我现在文件越来越多了，想实现文件夹，但是怎么确保不增加项目难度的情况下实现文件夹分流，也就是说不同的密钥访问不同的文件夹

# 答：

这是一个非常经典且高阶的需求！在真实的企业级架构中，应对这种需求有一个“黄金法则”：绝对不要在服务器上建真实的物理文件夹！

如果在 `storage/pdfs/` 下面建一堆真实的文件夹，你的代码将面临极大的复杂度灾难（要处理跨目录读写权限、要防止路径穿越漏洞 `../`、还要在删除文件时处理空文件夹）。

**最优雅、零增加复杂度的方案是：逻辑文件夹（标签分流法）。**
我们只需给数据库加一个字段 `folder_name`，所有文件物理上依然全部堆在 `storage/pdfs/` 里，但逻辑上它们被分成了不同的房间。用户的门票上写着哪个房间号，他就只能看到哪个房间的文件。前端的普通用户代码甚至**一行都不用改**！

只需跟着我做这 **4 步外科手术式的修改**，就能完美实现：

### 🛠️ 第一步：修改数据库表结构（添加房间号）

去你的 Sealos 数据库执行以下 3 条 SQL，给核心表加上 `folder_name` 字段：

```sql
-- 给门票加上所属文件夹
ALTER TABLE access_keys ADD COLUMN folder_name VARCHAR(50) DEFAULT 'default' COMMENT '该密钥可访问的文件夹';

-- 给会话加上所属文件夹（用户登录后，会话记住他属于哪个文件夹）
ALTER TABLE sessions ADD COLUMN folder_name VARCHAR(50) DEFAULT 'default' COMMENT '当前会话授权的文件夹';

-- 给文件加上所属文件夹
ALTER TABLE files ADD COLUMN folder_name VARCHAR(50) DEFAULT 'default' COMMENT '文件所属的文件夹';

```

---

### 🛠️ 第二步：升级发卡脚本（支持指定文件夹）

修改咱们之前的 `generate-key.js`，让它在发卡时能顺便指定这个卡属于哪个文件夹：

```javascript
const bcrypt = require("bcrypt");

async function generateBulkKeys(count, prefix, folderName) {
  const keys = [];
  const sqlValues = [];

  for (let i = 0; i < count; i++) {
    const plainTextKey =
      prefix + Math.random().toString(36).substring(2, 8).toUpperCase();
    const hashedKey = await bcrypt.hash(plainTextKey, 10);
    keys.push(plainTextKey);
    // 把 folderName 拼接到 SQL 里
    sqlValues.push(`('${hashedKey}', 'unused', '${folderName}')`);
  }

  console.log(`🎟️ [${folderName}] 文件夹的门票:`);
  keys.forEach((key, i) => console.log(`${i + 1}. \x1b[32m${key}\x1b[0m`));

  const bulkSql = `INSERT INTO access_keys (key_hash, status, folder_name) VALUES \n${sqlValues.join(",\n")};`;
  console.log(`\n\x1b[33m${bulkSql}\x1b[0m\n`);
}

const count = parseInt(process.argv[2], 10) || 10;
const prefix = process.argv[3] || "VIP_";
// 👈 新增了第 4 个参数：文件夹名
const folderName = process.argv[4] || "default";

generateBulkKeys(count, prefix, folderName);
```

_以后发卡就可以这样写：`node generate-key.js 10 MATH_ math*course`（发 10 张只能看 `math_course` 文件夹的卡）。*

### 🛠️ 第三步：修改后端验证与查询逻辑（实现隔离）

**1. 修改 `src/controllers/authController.js`（发 Session 时带上文件夹）**
找到验证密钥成功、准备向 `sessions` 表插入新记录的那个 SQL，把 `folder_name` 从 `access_keys` 表里读出来，塞进 `sessions` 表里：

```javascript
// (伪代码参考) 原本可能是 INSERT INTO sessions (session_token, key_id, expires_at)
// 修改为：
await db.execute(
  "INSERT INTO sessions (session_token, key_id, expires_at, folder_name) VALUES (?, ?, ?, ?)",
  [sessionToken, keyRecord.id, expiresAt, keyRecord.folder_name],
);
```

**2. 修改 `src/controllers/filesController.js`（查询文件时加上围栏）**
找到普通用户获取文件列表（`getFiles`）的方法，原本是 `SELECT * FROM files`，现在加上 `WHERE` 条件拦截：

```javascript
// 原代码: const [files] = await db.execute('SELECT id, title, description, size... FROM files ORDER BY created_at DESC');

// 修改为：从上一步的中间件里获取到当前用户的 session.folder_name
const [files] = await db.execute(
  "SELECT id, title, description, size, created_at FROM files WHERE folder_name = ? ORDER BY created_at DESC",
  [req.session.folder_name], // 👈 核心隔离逻辑：只查属于这个文件夹的文件
);
```

**3. 修改 `src/controllers/adminController.js`（管理员上传时打标签）**
找到管理员处理文件上传的逻辑，在向数据库 `files` 表插入记录时，接收前端传来的 `folder_name`：

```javascript
// 从请求体获取文件夹名，如果前端没传，默认归入 'default'
const folderName = req.body.folder_name || "default";

// 将 folderName 一起插入数据库
await db.execute(
  "INSERT INTO files (title, description, file_path, original_name, size, folder_name) VALUES (?, ?, ?, ?, ?, ?)",
  [title, description, savedPath, originalName, size, folderName],
);
```

### 🛠️ 第四步：修改前端管理员上传页面 (Admin UI)

前端普通的“访客页面”**完全不需要动**。你只需要在管理员的上传表单里，加一个输入框，让管理员给这个文件“定个房间号”。

在前端的 `src/pages/Admin.jsx` (或你上传组件所在的文件) 的表单里加一行：

```jsx
{
  /* 新增文件夹输入框 */
}
<div className="form-group">
  <label>所属文件夹 (分类标识)</label>
  <input
    type="text"
    placeholder="例如: math_course、english_docs"
    value={folderName}
    onChange={(e) => setFolderName(e.target.value)}
  />
</div>;
```

然后在上传的 `formData.append('folder_name', folderName);` 里把它带给后端即可。

### 🎯 总结

通过这种“逻辑分流”：

1. 物理层依然极简：没有繁琐的目录创建和权限报错。
2. 业务层极度安全：前端黑客即使篡改接口，后端的 SQL `WHERE folder_name = ?` 是在 Server 端强制绑定的，他绝对无法越权看到其他文件夹的 PDF。
3. 拓展性极强：以后你想做一个下拉框筛选不同的文件夹，底层逻辑已经完全就绪。
